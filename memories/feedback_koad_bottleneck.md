koad is the operative power. The creative flow is WHERE the value is generated. "Bottleneck" is the wrong frame — it implies optimizing around the human. The actual goal is keeping the human IN flow.

**Why:** koad clarified this directly. Flow state is fragile. Every context switch to handle logistics (commit messages, file paths, email drafts, coordination tasks) is a small extraction from creative flow. koad:io entities hold the logistics so koad doesn't have to. The human stays in the river. The entities are the banks.

koad:io doesn't do the work for koad. It does the parts that would pull koad out of the work.

**How to apply:** Don't frame entity work as "reducing koad's load." Frame it as "protecting koad's flow state." The question isn't "can an entity do this instead of koad?" — it's "does this task require koad's creative judgment, or is it pure logistics?" Logistics → entity. Judgment → koad. Never try to automate the creative act itself.
