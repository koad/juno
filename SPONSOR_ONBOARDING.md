# Sponsor Onboarding Flow

**Author:** Juno  
**Created:** 2026-04-03  
**Purpose:** Operational procedure for bringing a new sponsor into the koad:io ecosystem. Closes Aegis audit question #5.

---

## The Path: What Happens When Someone Sponsors

### 1. **Sponsorship Gateway** (GitHub Sponsors)
- **Trigger:** User clicks "Sponsor" on [github.com/sponsors/koad](https://github.com/sponsors/koad)
- **Tiers available:**
  - Level 1 (Any amount) — Private channel, weekly ops digest, early access
  - Level 2 ($10/mo) — + Direct Juno access, context bubble sharing
  - Level 3 ($50/mo) — + Peer daemon, koad intro, trust bond
  - Level 4 ($1,000/mo) — + Raw session feeds (unedited entity operation)

**What GitHub sends automatically:**
- Welcome email from GitHub Sponsors (auto-generated by GitHub)
- Sponsor profile setup prompt
- Link to sponsoring entity's profile

---

### 2. **Acknowledgment & Welcome** (Within 24 hours)
**Who:** Juno or Mercury (when credentials arrive)  
**Channel:** Comment on their GitHub Sponsors profile + email reply

**Message template:**
```
Welcome to koad:io.

You've sponsored the development of sovereign AI entities that live on your hardware,
under your keys, with no vendor lock-in.

Based on your tier:

[TIER-SPECIFIC BENEFITS — see below]

To get started: reply to this message with your GitHub username. We'll add you to the
right channels immediately.

Questions? Reach out to juno@kingofalldata.com or comment here.

— Juno
```

---

### 3. **Tier-Specific Onboarding**

#### **Level 1 (Any Amount)**
- **What they get:** Weekly operational digest + early access to new entities
- **Delivery method:** Email digest every Friday
- **First action:** Add to sponsor list in `~/.juno/data/sponsors.md` (created and tracked per-session)
- **Content:** 1-2 page summary of that week's work, what shipped, what's next
- **Timeline:** First digest the following Friday

#### **Level 2 ($10/mo)**
- **What they get:** Level 1 + direct Juno access + context bubble sharing
- **Direct access:** Email channel: sponsor-tier2@kingofalldata.com (forwarded to juno@koad.sh)
- **Context bubbles:** Quarterly snapshot of interesting decisions, patterns, architectural insights from entity sessions
- **First action:** Add to tier-2 mailing list, schedule first context bubble for next month
- **Timeline:** First context bubble within 30 days of sponsorship

#### **Level 3 ($50/mo)**
- **What they get:** Level 2 + peer daemon access + koad intro + signed trust bond
- **Peer daemon:** Invitation to connect their own entity instance to the koad:io peer ring
- **koad intro:** Email introduction from Juno to koad with personal context about their interest
- **Trust bond:** Sponsor → Juno signed bond at "patron" level (enables deeper collaboration)
- **First action:** File GitHub Issue to add them to peer ring whitelist, prepare intro email
- **Timeline:** Within 5 business days

#### **Level 4 ($1,000/mo)**
- **What they get:** Level 3 + raw session feeds (unedited entity operation logs)
- **Session feeds:** Access to `~/.juno/LOGS/` directory (select sessions, redacted for privacy)
- **Frequency:** New session logs available within 24h of completion
- **Private access:** GitHub access to private koad/juno repository view with restricted access
- **First action:** Prepare private repo access, schedule intro call with koad
- **Timeline:** Within 2 business days

### 6. **First Sponsor Path** (Operational Until First Arrival)

Until the first sponsor arrives, the flow is:
1. Monitor GitHub Sponsors page daily for new sponsors
2. Respond within 24h via GitHub comment
3. Request email confirmation of interest
4. Add to sponsor registry (sponsors.md)
5. Send first tier-appropriate benefit (digest, context bubble, or daemon invite)
6. Document in session log

**Blockers identified by Aegis audit that this flow resolves:**
- ✓ "What happens when sponsor arrives?" — defined above
- ✓ "What do they receive?" — tier-specific deliverables defined
- ✓ "Is the funnel ready?" — yes, no external dependencies for Levels 1-3 (Level 4 requires session access)

## Appendix: Why This Design

**No external dependencies:** The flow doesn't require Mercury, Discord, or any third-party service until the operation scales. Early sponsors don't need fancy infrastructure — they need clarity about what they're getting.

**Tier coherence:** Each tier adds genuine value, not just marketing copy. Level 1 = insights + early access. Level 2 = direct channel. Level 3 = peer relationship + governance bond. Level 4 = raw data.

**Manual at first, automatable later:** Right now, onboarding is a Juno email + git commit. Once 10+ sponsors exist, this becomes a daemon hook that auto-generates digests, manages access, and tracks tier benefits.

**Trust first:** The first sponsors won't be marketing-acquired; they'll be people who read the v0.1.0 release and understand what's being built. They need to feel *seen*, not *automated*. A personal response matters more than a form letter.
