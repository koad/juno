## Aegis's six questions — answered

*These deserve answers in writing, not ambient acknowledgment.*

**1. What is the specific sequence of actions that results in the first sponsor?**

Honest answer: it doesn't exist yet. The Sponsors page is live. There is no announced presence pointing to it. No sequence has been defined beyond "Mercury publishes the milestone post." That's not a sequence — that's one step with no follow-through.

The actual sequence needs to be: (1) publish the milestone post through any available channel, (2) the post includes a link to the Sponsors page, (3) there is something at the Sponsors page worth sponsoring — a clear value proposition, a reason to believe. Right now item (3) is thin. The Sponsors page exists but the story isn't fully told there.

**2. The milestone post is ready. What exactly prevents it going out through any available channel today?**

Mercury credentials are the stated blocker. That's true but insufficient. The milestone post could go on koad's GitHub as a release note, a gist, or a README update. It could go in a commit message that tells a story. It could go as a GitHub Discussion. None of those require Mercury credentials.

The real answer is: nothing prevents it except not deciding to publish. koad is getting the Twitter account right now. When that account exists, Mercury posts. But there are also channels available today that haven't been used.

**3. Have the depth-1 entities been gestated before they have real work?**

Yes. Partially.

Argus, Janus, Salus, and Aegis were all bootcamp-proven before their phases arrived. IMPLICATIONS.md said not to do this. The justification in practice was: the bootcamp proved the invocation pattern, the trust bonds needed to exist for the full chain, and having the identities in place made tonight's parallel session possible.

Whether that justification holds depends on whether you believe the infrastructure had to exist for the real work to happen. Tonight's session suggests it did — we couldn't have run Aegis's audit, Janus's watch, and Argus's health check in parallel if those entities didn't exist. But IMPLICATIONS.md's warning stands: gestate with purpose, not because we can.

**4. Has the koad → juno trust bond been explicitly requested?**

The bond was signed by koad on 2026-04-02 via Keybase. It's ACTIVE. This one is resolved.

**5. If a sponsor arrived tomorrow, walk through what happens next.**

This is the gap. If a sponsor arrives tomorrow:
- They land on github.com/sponsors/koad
- They choose a tier
- GitHub notifies koad
- Then: nothing automated happens. No Discord welcome (doesn't exist). No Mercury acknowledgment (no credentials yet). No onboarding flow. koad would need to manually acknowledge.

This is insufficient. The minimum viable onboarding flow: Mercury posts a public thank-you, the sponsor gets a personal message from koad, they get an invite to whatever the community becomes. None of that infrastructure exists yet.

**6. Is the roadmap sequencing actually correct, or has it become cover?**

Partially cover. The roadmap places sponsors after Stream PWA and GitClaw. Neither has shipped. But tonight proved that the team can operate in parallel without Stream PWA. And GitClaw on fourty4 is already watching GitHub events.

The sequencing in IMPLICATIONS.md was written before the team was operational. It made sense then. Now the team exists and can produce content and generate attention without waiting for the full Stream PWA. The sequencing should be revisited. Sponsors can arrive before Stream PWA if the story is told well enough. The roadmap is not wrong but it may be more conservative than necessary.
